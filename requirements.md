# 需求文档

## 简介

个性化旅游系统是一个创新的综合旅游平台，解决传统旅游系统无法提供的实际需求。系统涵盖旅游前的智能推荐、旅游中的多策略路线规划（包括室内导航）、场所查询、以及旅游后的多媒体日记生成与交流。系统基于自主设计的数据结构和算法，提供高性能的个性化服务，支持景区和校园内部导航、美食推荐、AIGC内容生成和社交分享功能。

## 术语表

- **系统（System）**: 个性化旅游系统
- **用户（User）**: 使用系统的旅游者
- **景区（Scenic_Area）**: 旅游目的地，包括景区和校园
- **景点（Attraction）**: 景区内的建筑物或参观点
- **服务设施（Service_Facility）**: 景区内的商店、饭店、洗手间等设施
- **美食（Food）**: 景区内的餐饮选项，包括饭店和窗口
- **菜系（Cuisine）**: 美食的分类，如川菜、粤菜等
- **道路图（Road_Graph）**: 景区内部的道路网络图，包含建筑物和设施的连接关系
- **拥挤度（Congestion_Factor）**: 道路拥挤程度，小于等于1的正数，真实速度=拥挤度×理想速度
- **交通工具（Transportation）**: 移动方式，包括步行、自行车、电瓶车
- **自行车道（Bicycle_Path）**: 校区内自行车专用道路
- **电瓶车路线（Electric_Cart_Route）**: 景区内电瓶车固定路线
- **室内结构（Indoor_Structure）**: 建筑物内部结构，包括大门、电梯、楼层和房间
- **旅游路线（Tour_Route）**: 景区内的参观路径
- **旅游日记（Travel_Diary）**: 用户根据旅游经历生成的多媒体记录
- **热度（Popularity）**: 景区或日记的访问量或受欢迎程度
- **评价（Rating）**: 用户对景区、美食或日记的评分
- **用户兴趣（User_Interest）**: 用户的旅游偏好和兴趣标签
- **美食爱好者（Foodie）**: 优先推荐美食丰富的景区和餐饮点的用户类型
- **摄影打卡者（Photographer）**: 优先推荐拍照热门景点的用户类型
- **历史文化爱好者（Culture_Enthusiast）**: 优先推荐历史文化景点的用户类型
- **自然风光爱好者（Nature_Lover）**: 优先推荐自然景观的用户类型
- **运动健身者（Sports_Enthusiast）**: 优先推荐徒步路线和运动场所的用户类型
- **休闲放松者（Relaxation_Seeker）**: 优先推荐安静舒适景点的用户类型
- **社交分享者（Social_Sharer）**: 关注热门打卡点和社交互动的用户类型
- **热度榜（Popularity_Ranking）**: 基于访问量排序的排行榜
- **评分榜（Rating_Ranking）**: 基于用户评分排序的排行榜
- **评价榜（Review_Ranking）**: 基于评价数量和质量排序的排行榜
- **个人兴趣榜（Personalized_Ranking）**: 综合考虑热度、评分、评价和用户兴趣的排行榜
- **拍照位置（Photo_Spot）**: 景点内推荐的最佳拍照位置
- **美食地图（Food_Map）**: 显示景区内所有美食位置的地图
- **实时热点（Trending）**: 当前最热门的景点和话题
- **签到（Check_in）**: 用户在景点的位置签到记录
- **成就徽章（Achievement_Badge）**: 用户完成特定任务获得的徽章
- **探索模式（Exploration_Mode）**: 推荐用户未尝试过类型的推荐模式
- **惊喜推荐（Surprise_Recommendation）**: 推荐与用户兴趣不完全匹配但可能喜欢的内容
- **浏览历史（Browsing_History）**: 用户的浏览记录和交互行为
- **推荐算法（Recommendation_Algorithm）**: 基于自主设计数据结构的排序算法
- **查找算法（Search_Algorithm）**: 基于自主设计数据结构的查询算法
- **排序算法（Sorting_Algorithm）**: 基于自主设计数据结构的排序算法
- **压缩算法（Compression_Algorithm）**: 无损压缩算法
- **AIGC引擎（AIGC_Engine）**: 人工智能生成内容引擎

## 需求

### 需求 1: 景区数据管理

**用户故事:** 作为系统管理员，我希望系统能够存储和管理大量景区数据，以便为用户提供丰富的旅游选择。

#### 验收标准

1. 系统应当存储至少200个景区数据
2. 对于每个景区，系统应当存储至少20个景点信息
3. 对于每个景区，系统应当存储至少10种服务设施类型
4. 对于每个景区，系统应当存储至少50个服务设施实例
5. 对于每个景区，系统应当存储包含至少200条边的道路图

### 需求 2: 用户账户管理

**用户故事:** 作为用户，我希望能够注册和登录系统，以便使用个性化功能。

#### 验收标准

1. 当用户提交有效的注册信息时，系统应当创建新用户账户
2. 当用户提交有效的登录凭证时，系统应当验证身份并授予访问权限
3. 当用户提交无效的登录凭证时，系统应当拒绝访问并返回错误信息
4. 系统应当支持至少10个用户账户

### 需求 3: 用户兴趣类型细分管理

**用户故事:** 作为用户，我希望能够选择细分的兴趣类型，以便获得更精准的个性化推荐。

#### 验收标准

1. 当用户首次登录时，系统应当提示用户从7种兴趣类型中选择
2. 系统应当支持美食爱好者、摄影打卡者、历史文化爱好者、自然风光爱好者、运动健身者、休闲放松者、社交分享者类型
3. 系统应当允许用户选择多个兴趣类型
4. 当用户请求修改兴趣时，系统应当更新用户兴趣配置
5. 系统应当持久化存储用户兴趣数据
6. 对于美食爱好者类型，系统应当在路线规划时增加途经餐饮点的权重
7. 对于摄影打卡者类型，系统应当在推荐中增加拍照热门景点的权重
8. 对于历史文化爱好者类型，系统应当在推荐中增加历史文化景点的权重
9. 对于自然风光爱好者类型，系统应当在推荐中增加自然景观的权重
10. 对于运动健身者类型，系统应当在推荐中增加徒步路线和运动场所的权重
11. 对于休闲放松者类型，系统应当在推荐中增加安静舒适景点的权重
12. 对于社交分享者类型，系统应当在推荐中增加热门打卡点的权重

### 需求 4: 多维度排行榜系统

**用户故事:** 作为用户，我希望通过不同维度的排行榜查看景区推荐，以便根据不同需求选择旅游目的地。

#### 验收标准

1. 系统应当提供热度榜、评分榜、评价榜、个人兴趣榜四种排行榜
2. 当用户选择热度榜时，系统应当基于访问量使用排序算法返回前10个景区
3. 当用户选择评分榜时，系统应当基于用户评分使用排序算法返回前10个景区
4. 当用户选择评价榜时，系统应当基于评价数量和质量使用排序算法返回前10个景区
5. 当用户选择个人兴趣榜时，系统应当综合热度、评分、评价和用户兴趣计算推荐分数
6. 系统应当使用基于自主设计数据结构的推荐算法返回前10个景区
7. 推荐算法应当在不进行完全排序的情况下获得前10个最高分景区
8. 系统应当在个人兴趣榜中显示"已参考热度、评分、评价等因素"提示
9. 系统应当在推荐结果中显示景区的热度值和评价分数
10. 当用户未设置兴趣时，个人兴趣榜应当仅基于热度和评价进行推荐
11. 推荐算法应当支持数据动态变化的场景
12. 系统应当允许用户在不同排行榜之间切换

### 需求 4.1: 增量推荐系统

**用户故事:** 作为用户，我希望通过刷新获得新的推荐结果，以便发现更多感兴趣的景区。

#### 验收标准

1. 当用户首次请求推荐时，系统应当返回10个推荐结果
2. 当用户刷新推荐时，系统应当返回10个新的推荐结果
3. 系统应当标记已展示的推荐结果并在后续推荐中排除
4. 当用户收藏某个景区时，系统应当记录收藏行为
5. 当用户标记某个景区为不感兴趣时，系统应当在后续推荐中过滤类似景区
6. 系统应当记录用户的浏览历史和交互行为
7. 系统应当基于用户的收藏和不感兴趣行为动态调整推荐权重
8. 系统应当持久化存储用户的浏览历史和交互记录
9. 当已展示结果数量超过100个时，系统应当重置展示记录
10. 系统应当在300毫秒内返回刷新后的推荐结果

### 需求 4.2: 景区查询

**用户故事:** 作为用户，我希望通过多种方式查询景区和学校，以便快速找到目标地点。

#### 验收标准

1. 当用户输入景区名称时，系统应当使用查找算法返回匹配的景区列表
2. 当用户输入类别时，系统应当使用查找算法返回该类别的所有景区
3. 当用户输入关键字时，系统应当使用查找算法返回包含该关键字的景区列表
4. 系统应当支持按热度排序查询结果
5. 系统应当支持按评价排序查询结果
6. 排序功能应当使用基于自主设计数据结构的排序算法

### 需求 4.3: 摄影打卡功能

**用户故事:** 作为摄影打卡者，我希望获得景点的最佳拍照位置和时间推荐，以便拍摄高质量照片。

#### 验收标准

1. 当用户查看景点详情时，系统应当显示该景点的最佳拍照位置列表
2. 系统应当为每个拍照位置提供最佳拍照时间建议
3. 系统应当基于光线条件和人流量计算最佳拍照时间
4. 系统应当显示热门拍照角度的示例照片
5. 当用户上传打卡照片时，系统应当将照片添加到景点相册
6. 系统应当显示该景点的打卡人数统计
7. 系统应当显示该景点的热门照片排行
8. 系统应当基于照片点赞数和浏览量计算热门照片排名
9. 系统应当为摄影打卡者类型用户推荐AR滤镜和拍照特效
10. 当用户选择AR滤镜时，系统应当在相机界面应用该滤镜

### 需求 4.4: 美食爱好者专属功能

**用户故事:** 作为美食爱好者，我希望获得美食地图和美食路线规划，以便高效品尝景区美食。

#### 验收标准

1. 当用户选择景区后请求美食地图时，系统应当显示景区内所有美食位置
2. 系统应当在美食地图上标注饭店、窗口和美食摊位的位置
3. 当用户请求美食路线规划时，系统应当生成途经多个美食点的优化路线
4. 系统应当为美食爱好者类型用户生成美食打卡清单
5. 美食打卡清单应当包含至少10个推荐必吃美食
6. 当美食点营业时间临近或排队人数较多时，系统应当发送提醒通知
7. 系统应当基于营业时间和当前时间计算提醒时机
8. 系统应当提供美食组合推荐功能
9. 当用户选择某个美食时，系统应当推荐搭配的美食套餐
10. 系统应当基于菜系、口味和用户历史偏好计算美食组合推荐

### 需求 5: 景区详情查询

**用户故事:** 作为用户，我希望查看景区的详细信息，以便了解景区内容和规划行程。

#### 验收标准

1. 当用户选择某个景区时，系统应当显示该景区的基本信息
2. 系统应当显示景区内所有景点的列表
3. 系统应当显示景区内所有服务设施的列表和位置
4. 系统应当提供景区道路图的可视化展示

### 需求 6: 景区内路线规划 - 最短距离策略

**用户故事:** 作为用户，我希望在景区内规划距离最短的参观路线，以便减少行走距离。

#### 验收标准

1. 当用户选择起点和终点时，系统应当基于道路图计算距离最短路径
2. 系统应当在300毫秒内返回路线规划结果
3. 系统应当显示路线的总距离
4. 当起点和终点之间不存在路径时，系统应当返回提示信息

### 需求 6.1: 景区内路线规划 - 最短时间策略

**用户故事:** 作为用户，我希望在景区内规划时间最短的参观路线，以便节省旅游时间。

#### 验收标准

1. 当用户选择起点和终点时，系统应当基于道路图和拥挤度计算时间最短路径
2. 系统应当使用公式"真实速度=拥挤度×理想速度"计算通行时间
3. 系统应当在300毫秒内返回路线规划结果
4. 系统应当显示路线的总时间和总距离
5. 当起点和终点之间不存在路径时，系统应当返回提示信息

### 需求 6.2: 校区内多交通工具路线规划

**用户故事:** 作为用户，我希望在校区内使用自行车和步行规划最短时间路线，以便高效到达目的地。

#### 验收标准

1. 当用户在校区内请求路线规划时，系统应当考虑自行车和步行两种交通工具
2. 系统应当确保自行车仅在自行车道上行驶
3. 系统应当假设任何地点都有自行车可用
4. 系统应当基于拥挤度计算不同交通工具的通行时间
5. 系统应当返回时间最短的路线，可以是多种交通工具混合
6. 系统应当显示路线中每段使用的交通工具

### 需求 6.3: 景区内多交通工具路线规划

**用户故事:** 作为用户，我希望在景区内使用步行和电瓶车规划最短时间路线，以便快速游览景区。

#### 验收标准

1. 当用户在景区内请求路线规划时，系统应当考虑步行和电瓶车两种交通工具
2. 系统应当确保电瓶车仅在固定电瓶车路线上行驶
3. 系统应当假设电瓶车上车即走
4. 系统应当基于拥挤度计算不同交通工具的通行时间
5. 系统应当返回时间最短的路线，可以是多种交通工具混合
6. 系统应当显示路线中每段使用的交通工具

### 需求 6.4: 室内导航 - 大门到电梯

**用户故事:** 作为用户，我希望在建筑物内从大门导航到电梯，以便找到正确的电梯位置。

#### 验收标准

1. 当用户在建筑物大门请求到达某楼层时，系统应当计算从大门到电梯的路径
2. 系统应当基于室内结构数据生成导航指引
3. 系统应当显示行走距离和预计时间
4. 当建筑物有多个电梯时，系统应当选择最近的电梯

### 需求 6.5: 室内导航 - 楼层间电梯导航

**用户故事:** 作为用户，我希望获得楼层间的电梯导航指引，以便到达目标楼层。

#### 验收标准

1. 当用户请求到达不同楼层时，系统应当生成电梯导航指引
2. 系统应当显示需要乘坐的电梯和目标楼层
3. 系统应当计算电梯等待和乘坐的预计时间
4. 当目标楼层不可达时，系统应当返回提示信息

### 需求 6.6: 室内导航 - 楼层内到房间

**用户故事:** 作为用户，我希望在楼层内导航到具体房间，以便找到目标位置。

#### 验收标准

1. 当用户在某楼层请求到达具体房间时，系统应当计算楼层内路径
2. 系统应当基于室内结构数据生成导航指引
3. 系统应当显示行走距离和预计时间
4. 当房间不存在时，系统应当返回提示信息

### 需求 6.7: 导航图形界面

**用户故事:** 作为用户，我希望通过图形界面查看地图和导航路径，以便直观理解路线。

#### 验收标准

1. 系统应当提供地图可视化展示功能
2. 当系统生成路线后，系统应当在地图上高亮显示路径
3. 系统应当在地图上标注起点、终点和途经点
4. 系统应当支持地图缩放和平移操作
5. 对于室内导航，系统应当显示建筑物平面图

### 需求 7: 多景点路线优化

**用户故事:** 作为用户，我希望规划访问多个景点的最优路线，以便减少行走距离。

#### 验收标准

1. 当用户选择多个景点时，系统应当计算访问所有景点的优化路线
2. 系统应当返回景点访问顺序和路径
3. 当景点数量超过10个时，系统应当在5秒内返回近似最优解
4. 系统应当显示总路线距离

### 需求 7.1: 智能时间规划

**用户故事:** 作为用户，我希望系统根据景点开放时间和拥挤度生成一日游计划，以便高效游览景区。

#### 验收标准

1. 当用户请求一日游计划时，系统应当基于景点开放时间生成行程
2. 系统应当基于用户兴趣类型选择推荐景点
3. 系统应当基于拥挤度预测安排景点访问顺序
4. 系统应当在行程中安排用餐时间和休息时间
5. 系统应当基于用户体力偏好调整行程强度
